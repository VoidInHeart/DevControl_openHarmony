# 基于 OpenHarmony 的家居设备智能控制系统——需求文档

| 文档属性 | 内容 |
| --- | --- |
| 文档版本 | V1.1 |
| 当前平台基线 | HarmonyOS 6.1.1 / API 24 |
| APP 架构 | ArkTS + ArkUI + Native C++ |
| 网关架构 | Python + FastAPI + HTTPS/WSS + SQLite |
| 编制日期 | 2026 年 7 月 |

## 一、项目目标

本项目面向“中国软件杯”A9 赛题，开发一套基于 OpenHarmony 生态的家居设备智能控制系统。系统通过 APP 与独立虚拟网关之间的真实加密网络链路，实现灯光、温湿度、空调、门锁和智能窗帘的远程查看、控制、联动及审计。

项目当前不接入真实主板、传感器或家电。所有设备由独立网关进程模拟，APP 不得通过本地函数直接伪造设备执行结果。

## 二、系统范围

系统由三部分组成：

1. **控制端 APP**：负责网关配对、设备展示、控制交互、离线反馈和重新配对。
2. **独立虚拟网关**：负责认证、命令校验、设备权威状态、自动化规则、品牌适配和审计日志。
3. **模拟设备**：包括客厅灯、环境传感器、客厅空调和入户门锁。

首版范围不包括：

- 真实家电私有协议或厂商认证；
- 公网多租户、账号中心和云端运维；
- 生产环境 PKI、商业门禁安全认证；
- OpenHarmony API 20 兼容交付。

## 三、功能需求

### 3.1 控制中心

| 编号 | 需求 |
| --- | --- |
| CC-01 | 用户可以输入 HTTPS 网关地址和一次性配对码完成配对。 |
| CC-02 | APP 展示连接中、在线、离线和已关闭状态；连接断开后返回安全配对页。 |
| CC-03 | 配对成功后通过 WSS 接收设备快照、状态变化、命令结果、告警和心跳。 |
| CC-04 | 网络中断时保留最后一次快照并禁止产生伪成功状态。 |
| CC-05 | 连接中断后禁止继续控制并返回安全配对页；保留上次网关地址，用户重新输入当前一次性配对码后拉取权威快照。 |
| CC-06 | 支持一键执行离家模式：关灯、关闭空调并锁门。 |

### 3.1.1 首页导航与页面复用

首页顶部标签应固定提供“总控”、动态房间标签和“设置”：房间标签由当前可展示设备所属的房间自动生成。总控页按功能聚合展示灯光、窗帘、环境和门锁；门锁仅在总控中展示，不进入房间设备列表。

用户从总控功能卡或房间设备列表进入设备后，必须进入同一套完整的设备控制页面，而非功能不一致的简化页面。相同类型设备应复用控制模块，并以设备 ID 绑定其当前状态与控制命令。

窗帘功能作为总控分类保留；在当前网关协议尚未接入窗帘设备前，应展示无可用设备状态，不应提供虚假的可控入口。

| 编号 | 需求 |
| --- | --- |
| DR-01 | 支持锁门和解锁，解锁前必须二次确认。 |

## 四、协议与状态需求

1. APP 与网关的业务通信仅使用 HTTPS/WSS，不提供 HTTP/WS 降级路径。
2. 网关是设备状态的唯一权威来源；APP 仅缓存快照。
3. 命令包含 `protocolVersion`、`messageId`、`deviceId`、`timestamp`、`nonce`、`action` 和 `expectedStateVersion`。
4. `messageId` 用于五分钟幂等窗口；重复消息返回原结果，不重复执行设备动作。
5. nonce 为每条消息独立生成的 96 位随机数；重复 nonce 或超出时间窗口的消息必须拒绝。
6. 网关维护递增 `stateVersion`；陈旧版本命令返回 `STATE_CONFLICT`。
7. APP 忽略低于本地已知版本的乱序状态事件。

## 五、安全需求

### 5.1 传输层

- 网关启动必须提供 TLS 证书和私钥；缺失时禁止退化为明文服务。
- APP 验证服务器证书，不允许跳过证书校验。
- 演示证书的 SAN 必须覆盖 APP 实际连接的主机名或 IP。

### 5.2 数据层

- Native C++ 使用安全随机数生成 messageId 和 nonce。
- 控制载荷使用 AES-256-GCM 加密，认证标签长度为 128 位。
- AAD 覆盖协议版本、消息编号、设备编号、时间戳、消息类型、动作和期望状态版本。
- 密文、认证标签或 AAD 被篡改时，网关不得执行命令。

### 5.3 密钥与凭据

- 源码、配置、数据库和日志不得包含硬编码明文密钥或凭据。
- APP 使用 HUKS 本地密钥加密配对凭据和会话数据密钥。
- HUKS 包装采用 AES-256-GCM；Preferences 只保存密文封装、系统安全随机生成的 96 位 IV、128 位认证标签和结构版本。解密时认证标签通过 HUKS `HUKS_TAG_AE_TAG` 独立传入。网关地址也位于受保护的密文中，HUKS 别名不属于秘密。
- APP 重启时可以解密恢复会话；明文材料只在当前进程内短暂使用。
- HUKS 不可用、密钥操作失败或 Preferences 写入失败时必须中止会话保存，不得降级为明文持久化。
- 用户重新配对时覆盖旧封装；配对失败不得破坏仍可用的旧会话。

### 5.4 配对保护

- 网关一次性配对码为六位数字，有效期五分钟。
- 配对成功或配对码到期后立即生成不同的新配对码，不能长期复用固定值。
- 同一来源连续五次失败后锁定十分钟。
- 网关仅保存客户端凭据摘要，日志不得输出明文凭据和数据密钥。
- 网关签发凭据时返回 `issuedAt` 和 `expiresAt`；默认有效期 24 小时且可配置。到期后 REST 和已建立的 WSS 均返回 `AUTH_FAILED`，APP 清除旧会话并重新配对。

## 六、非功能需求

| 编号 | 类别 | 要求 |
| --- | --- | --- |
| NF-01 | 架构 | APP 固定采用 ArkTS + Native C++；设备模拟必须位于独立网关。 |
| NF-02 | 平台 | 当前版本以 HarmonyOS 6.1.1 / API 24 为构建基线。 |
| NF-03 | 可用性 | 页面具备空数据、加载、处理中、成功、失败、离线和重新配对反馈，并记住上次网关地址。 |
| NF-04 | 可靠性 | 连接中断后回到配对页；重新配对成功后以全量快照校正状态。 |
| NF-05 | 可扩展性 | 新设备通过注册式驱动和受限能力描述扩展；常见新设备不修改既有领域核心或 APP 路由。 |
| NF-06 | 可审计性 | 安全操作记录时间、设备、动作、结果和错误原因，不记录敏感正文。 |
| NF-07 | 构建 | ArkTS 与 Native C++ 必须共同通过 Hvigor HAP 构建。 |

## 七、验收要求

### 7.1 当前自动化证据

- Python 网关现有 26 项 pytest 覆盖驱动注册、智能窗帘、虚拟风扇接入、一次性配对码轮换、凭据过期、已连接 WSS 失效、快照、Schema、幂等、状态冲突、自动化、离家场景、持久化和密文篡改拒绝。
- 真实 HTTPS/WSS 脚本已覆盖合法 CA 配对、灯光控制、错误证书、密文/AAD/时间戳/nonce/credential 负向测试。
- 2026-07-18 隔离端口回归的 1000 条命令成功率 100%，命令 P95 4.188 ms，状态事件 P95 4.538 ms。
- Debug HAP 已通过 ArkTS、Native C++ 和资源编译；签名演示 HAP 已生成，版本化签名口令已清除。
- 敏感信息扫描已确认不存在旧硬编码密钥、伪 TLS、APP 内设备模拟器、私钥正文或签名口令。

### 7.2 必须补充的设备端验证

- APP 安装、首次配对和四大功能页面回归；
- Native AES-256-GCM 与网关实际互通；
- APP 重启后 HUKS 会话恢复；
- 证书错误、网络中断和网关恢复；
- 门锁二次确认及自动上锁；
- Preferences、日志和数据库明文敏感信息检查；
- HarmonyOS 6.1.1 真机连续运行 30 分钟及资源增长记录。

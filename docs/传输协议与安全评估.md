# 传输协议与数据安全评估

评估日期：2026-07-18

## 一、结论

改进前，系统的 APP—网关链路已具备较完整的安全闭环，但实际只支持 HTTPS REST 和 WSS，两者都建立在 HTTP/TLS 技术栈上，协议覆盖不宜评价为“广”。

本轮改进后，网关具备四种可运行的安全传输方式：HTTPS REST、WSS、HTTPS/SSE 和 MQTT 5 over TLS。所有命令入口复用同一套 Bearer 会话、AES-256-GCM、时间窗口、nonce 防重放、messageId 幂等、stateVersion 冲突检测和审计链路。MQTT 默认关闭，只有在配置可信 CA 以及 mTLS 或用户名/密码认证后才能启动，不存在明文 MQTT 回退。

当前结论为：项目已满足竞赛级智能家居原型对主流请求响应、实时双向、单向事件流和物联网发布订阅传输的覆盖要求，数据传输安全性较高；但不等同于生产级商用认证，也不宣称支持 CoAP/DTLS、Matter、Zigbee、蓝牙或真实厂商私有协议。

## 二、协议覆盖

| 传输方式 | 状态 | 适用场景 | 安全措施 |
| --- | --- | --- | --- |
| HTTPS REST | 默认启用 | 配对、快照、历史、日志、加密命令 | TLS 1.2+、证书校验、Bearer、命令 AES-GCM |
| WSS | 默认启用 | 双向实时命令、状态、告警、心跳 | TLS 1.2+、证书校验、Bearer、命令 AES-GCM |
| HTTPS/SSE | 默认启用 | 只需服务端推送的轻量客户端或 WSS 受限网络 | TLS 1.2+、Bearer、15 秒心跳、断线重连语义 |
| MQTT 5 over TLS | 代码可用、默认关闭 | 接入已有安全 MQTT Broker 的发布订阅系统 | TLS 1.2+、主机名/CA 校验、mTLS 或账号认证、命令 Bearer + AES-GCM、QoS 1 |
| CoAP over DTLS | 未支持 | 资源受限 UDP 终端 | 当前无 DTLS 运行栈和受限终端，不作兼容声明 |

健康接口 `GET /api/v1/health` 会返回传输能力及 MQTT 当前是否启用。MQTT 是网关到外部 Broker 的安全桥接能力，不包含内置 Broker；生产部署必须在 Broker 上配置最小权限 ACL，限制命令、结果、设备状态和事件主题的发布/订阅范围。

## 三、安全评估

### 3.1 传输层

- 业务服务缺少证书或私钥时拒绝启动，不回退到 HTTP/WS。
- 服务端 SSLContext 显式设置最低 TLS 1.2，允许 TLS 1.3，限制为 ECDHE + AEAD 密码套件并关闭 TLS 压缩。
- APP 只接受 `https://` 网关地址，WSS 固定由安全地址派生，证书校验不可跳过。
- MQTT 始终使用校验主机名的 TLS 客户端上下文；未提供 Broker CA 或客户端认证时配置校验失败。
- HTTP、WSS 和 MQTT 单消息上限默认 64 KiB；加密正文上限 32 KiB，降低超大消息造成的资源耗尽风险。

### 3.2 会话与消息层

- 六位一次性配对码五分钟轮换，同一来源五次失败锁定十分钟。
- Bearer 凭据默认 24 小时到期；REST、SSE、WSS 和 MQTT 命令均重新校验会话。
- 命令正文使用 AES-256-GCM；AAD 覆盖协议版本、消息编号、设备、时间戳、类型、动作和状态版本。
- 96 位随机 nonce、30 秒时间窗口、五分钟重放/幂等窗口共同阻止篡改与重复执行。
- MQTT 载荷中的命令仍为 AES-GCM 密文；客户端凭据仅用于网关鉴权，不写入结果主题、事件正文、SQLite 或审计正文。

### 3.3 本地密钥与审计

- APP 使用 HUKS AES-256-GCM 包装会话数据，Preferences 不保存明文凭据和数据密钥。
- 网关只保存凭据摘要，安全操作记录设备、动作、结果和错误码，不记录控制明文或密钥。
- HTTP API 返回 `no-store`、HSTS、`nosniff` 和 `no-referrer` 安全响应头。

## 四、验证证据

在 `VirtualGateway` 目录执行：

```powershell
python -m pytest -q
```

当前结果为 `30 passed`。新增覆盖包括：

- HTTPS 加密命令与 WSS 命令复用同一处理链；
- SSE Bearer 会话、心跳和状态广播；
- MQTT 5 TLS 上下文、Broker 认证前置条件、QoS 1 命令/结果、保留状态主题；
- 无效 MQTT 凭据不执行设备动作且不发布结果；
- TLS 最低版本为 1.2，旧 RC4/3DES 套件不可用；
- 服务端消息尺寸限制和 APP 明文地址拒绝逻辑进入构建检查范围。

## 五、剩余边界与后续建议

1. MQTT 上线前应使用真实 Broker 完成证书轮换、ACL 越权、断线重连、离线消息和集群故障演练；当前自动化使用协议客户端替身验证网关桥接逻辑。
2. 演示 CA 和演示证书只能用于开发；生产环境需接入正式 PKI、吊销/轮换机制和安全运维流程。
3. 若后续接入低功耗受限设备，再独立实现 CoAP over DTLS 1.2/1.3，并补齐设备身份、资源授权、重传去重和真实终端互操作测试。
4. Matter、Zigbee、BLE 和品牌私有协议属于设备接入协议或链路技术，不应与本轮 APP—网关传输能力混为一谈；只有完成真实 SDK/硬件互操作后才能列为支持。

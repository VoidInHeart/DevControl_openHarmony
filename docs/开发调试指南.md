# DevControl 开发调试与全链路跑通指南

| 项目 | 内容 |
| --- | --- |
| 适用版本 | DevControl V1.1 |
| APP 基线 | HarmonyOS 6.1.1 / API 24，ArkTS + Native C++ |
| 网关基线 | Python 3.11～3.13，FastAPI + HTTPS/WSS + SQLite |
| 推荐主机 | Windows 10/11 + DevEco Studio |
| 更新日期 | 2026-07-17 |

## 一、目标与运行拓扑

本指南用于让首次接触项目的开发者从干净环境跑通以下链路：

1. 启动独立 FastAPI 虚拟网关。
2. 使用演示 CA 验证 HTTPS/WSS。
3. 在 DevEco Studio 构建并安装 HarmonyOS APP。
4. 完成健康检查、六位码配对、HUKS 会话保存和 WSS 连接。
5. 验证灯光、环境/空调、门锁、离家模式和诊断页面。
6. 验证离线回配对页、地址记忆、故障注入和自动化测试。

正式运行链路如下：

```text
HarmonyOS APP
  ├─ HTTPS 8443：配对、快照、日志、环境历史
  ├─ WSS   8443：安全命令、状态事件、报警、心跳
  ├─ HUKS：保护 credential、dataKey 和网关地址
  └─ Native C++：安全随机数、AES-256-GCM、消息 ID、日志脱敏
          │
          ▼
FastAPI 权威网关
  ├─ 四类模拟设备与自动化规则
  ├─ SQLite 审计、报警和环境历史
  └─ 127.0.0.1:18444 本机维护接口
```

网关是唯一设备状态权威。APP 点击控件后只显示“处理中”，最终状态必须以网关返回的更高 `stateVersion` 为准。

## 三、准备开发环境

### 3.1 必需软件

- DevEco Studio，包含 HarmonyOS 6.1.1 / API 24 SDK。
- Python 3.11～3.13。
- PowerShell。
- API 24 手机模拟器，或 HarmonyOS 6.1.1 真机。
- 电脑与测试设备处于可互通的局域网。

确认 Python：

```powershell
python --version
python -c "import sys; print(sys.executable)"
```

### 3.2 推荐使用独立 Python 环境

推荐为项目创建虚拟环境，避免 Anaconda 全局包和用户级包混用：

```powershell
cd C:\Users\admin\Desktop\openHarmonyProject\VirtualGateway
python -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install --upgrade pip
python -m pip install -r .\requirements.txt
```

如果 PowerShell 阻止当前终端执行激活脚本，可仅对当前进程放开：

```powershell
Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass
.\.venv\Scripts\Activate.ps1
```

也可以不激活，直接使用：

```powershell
.\.venv\Scripts\python.exe -m pip install -r .\requirements.txt
```

若现有 Python 环境已经通过网关测试，不要求为了本次调试重新安装。

### 3.3 pip 代理错误

出现以下内容表示 pip 使用了失效代理，而不是依赖本身错误：

```text
ProxyError
WinError 10061
Cannot connect to proxy
```

先在发生错误的同一个 DevEco Terminal 中检查：

```powershell
Get-ChildItem Env:*proxy*
python -m pip config debug
```

如果确认代理已失效，只清理当前 Terminal 的代理变量：

```powershell
Remove-Item Env:HTTP_PROXY -ErrorAction SilentlyContinue
Remove-Item Env:HTTPS_PROXY -ErrorAction SilentlyContinue
Remove-Item Env:ALL_PROXY -ErrorAction SilentlyContinue
Remove-Item Env:PIP_PROXY -ErrorAction SilentlyContinue
```

然后重新执行安装。不要在不清楚团队网络配置时修改系统级代理。

执行：

```powershell

```

选择正在使用且带默认网关的 WLAN 或以太网 IPv4，例如：

## 八、启动网关

在独立的 DevEco Terminal 标签页中运行：

```powershell
cd C:\Users\admin\Desktop\openHarmonyProject\VirtualGateway
.\scripts\run_gateway.ps1
```

保持该终端运行。启动时会输出：

- 六位配对码及剩余有效时间；
- 仅用于本机维护接口的 Admin Token；
- HTTPS/WSS 服务端口 `8443`；
- 维护端口 `127.0.0.1:18444`。

不要复制、截图或粘贴 Admin Token。Token 泄露后按 `Ctrl+C` 停止网关并重新启动，使其轮换。

Windows 防火墙首次询问时，只允许 Python 访问专用网络。不要把维护端口开放到局域网。

检查端口：

```powershell
netstat -ano | Select-String ':8443'
netstat -ano | Select-String ':18444'
```

正常情况下 `8443` 监听所有网卡，`18444` 只监听 `127.0.0.1`。

## 九、验证 HTTPS 健康状态

推荐使用项目已经安装的 `httpx`。该命令执行 CA、证书有效期和主机名校验，同时忽略终端中的代理变量：

```powershell
python -c "import httpx; r=httpx.get('https://${GatewayIp}:8443/api/v1/health', verify=r'.\certs\demo-ca.crt', trust_env=False); print(r.status_code, r.json())"
```

预期：

```text
200 {'status': 'ok', 'protocolVersion': '1.0', ...}
```

Windows `curl.exe` 备选命令：

```powershell
curl.exe --noproxy "*" --ssl-no-revoke --cacert ".\certs\demo-ca.crt" "https://${GatewayIp}:8443/api/v1/health"
```

说明：

- `--noproxy "*"` 只让本次请求不经过本机代理；
- `--ssl-no-revoke` 用于演示私有 CA 没有公开 CRL 的 Schannel 环境；
- CA 和主机名仍会校验；
- 禁止使用 `-k` 或 `--insecure`，否则失去证书验证意义。

PowerShell 自动显示的 `>>` 是续行提示符，不属于命令，不要复制。

## 十二、完成首次配对

APP 配对页填写：

```text
网关地址：https://<电脑局域网 IP>:8443
配对码：<网关启动终端当前显示的六位码>
```

示例：

```text
https://192.168.31.150:8443
```

必须手动覆盖页面中的示例地址，不要填写：

- `http://...`；
- 手机或模拟器自己的 `127.0.0.1`；
- 证书 SAN 中不存在的 IP；
- 维护端口 `18444`。

成功流程：

```text
HTTPS 健康检查
  → POST /api/v1/pair
  → HUKS 首次生成包装密钥
  → AES-256-GCM 加密保存会话
  → GET /api/v1/devices
  → WSS /ws/v1/events
  → APP 显示在线
```

配对码有效期为5分钟，配对成功或到期后立即轮换。同一来源连续五次失败会锁定10分钟。不要复用已经成功使用的配对码，也不要反复尝试错误配对码。

## 十三、HUKS 首次运行验证

干净安装时，包装密钥本来就不存在。部分 API 24 模拟器会让 `isKeyItemExist()` 抛出：

```text
12000011: The entity does not exist
```

当前 `GatewaySessionStore` 已把 `12000011` 作为正常的“密钥不存在”分支，随后调用 `generateKeyItem()`；其他 HUKS 错误仍会失败且不会降级成明文。AES-GCM 解密把 16 字节认证标签通过 `HUKS_TAG_AE_TAG` 独立传入；`finishSession()` 失败后的 `abortSession()` 也不会覆盖原始错误。

首次配对验证标准：

1. 不再向用户显示 `12000011`。
2. 配对完成后进入主页面并显示网关在线。
3. 关闭并重新打开 APP，网关不重启时能够恢复会话。
4. Preferences 中只存在版本、IV、密文和认证标签组成的加密封装。
5. 日志中不存在完整 credential、dataKey、nonce 或 Admin Token。

如果仍然出现 `12000011`：

1. 确认已经拉取包含 `HUKS_ITEM_NOT_EXIST` 的最新 `GatewaySessionStore.ets`。
2. 在 DevEco 执行 Clean、Rebuild。
3. 卸载模拟器中的旧 APP，再重新 Run，排除安装了旧 HAP。
4. 不要优先清空整个模拟器，也不要关闭 HUKS 或改成明文保存。

网关会话只保存在内存，且客户端凭据默认有效24小时。网关重启或凭据到期后，APP 会清除旧会话并回到配对页；这与 APP 自身在凭据有效期内重启时的 HUKS 恢复是两种不同场景。

## 十四、业务功能冒烟顺序

### 14.1 总控中心

- 显示网关在线和五台设备。
- 执行离家模式。
- 确认灯关闭、空调关闭、门锁上锁。
- 查看逐设备成功或失败结果。

### 14.1.1 首页导航、总控与房间回归

1. 启动应用后确认顶部显示“总控”、实际房间名称和“设置”，不再把灯光、环境等功能作为顶部标签。
2. 在“总控”确认“我的全屋”紧接顶部标签区显示，功能卡包含灯光、窗帘、环境、门锁；窗帘在未接入协议时显示空状态。
3. 选择任一房间，确认只显示该房间设备且不包含门锁；点击设备后应进入完整的类型控制页面。
4. 分别从总控/房间进入同一盏灯，验证开关与滑块即时同步；从总控进入门锁并点击解锁，验证会显示远程解锁确认。
5. 若新增房间或设备类型，优先在 `DeviceRegistry` 补充房间、能力和功能分类；页面入口会据此生成，避免新增平行控制页。

### 14.2 灯光中心

- 开关灯。
- 调节 0～100 亮度。
- 图标始终保持固定大小，状态只改变透明度。
- 关闭透明度为低值；开启后透明度随亮度明显变化。
- 验证自动照明、照度阈值、无人延时和手动优先。

### 14.3 环境与空调

- 环境温湿度、照度和有人状态约每2秒更新。
- 查看24小时趋势。
- 测试空调电源、制冷/制热/除湿模式。
- 调节 16～30℃目标温度。
- 切换 `haierSim`、`greeSim`、`mideaSim`。

### 14.4 门锁

- 解锁必须二次确认。
- 解锁后等待自动上锁。
- 配置自动上锁开关和延时。
- 查看低电量、卡锁、离线提示。
- 查看不可由家庭用户清除的审计记录。

### 14.5 设置与诊断

- 查看网关地址、协议、证书信任方式和最后同步时间。
- 查看脱敏审计日志和错误码说明。
- 验证清除会话并重新配对。

## 十五、离线与恢复验证

### 15.1 仅中断网络

保持网关进程运行，临时断开设备网络：

- APP 应显示离线并返回安全配对页；
- 所有控制入口禁用；
- 保留最后同步时间和缓存快照；
- 配对页自动填入上一次使用的网关地址；
- 网络恢复后输入网关当前一次性配对码，重新配对并获取全量设备快照。

### 15.2 停止网关

在网关终端按 `Ctrl+C`：

- APP 应进入离线状态并返回配对页；
- 设备卡片不得伪造成功；
- 待处理命令应超时并显示协议错误码。

重新启动网关后，因网关内存会话已失效，需要使用新配对码重新配对；上次网关地址会保留。

## 十六、维护接口与故障注入

维护接口只允许在网关电脑本机使用。把网关启动时输出的 Token 仅复制到本机变量：

```powershell
$AdminToken = Read-Host "Admin Token"
$Headers = @{ "X-Admin-Token" = $AdminToken }
```

查询当前配对码：

```powershell
Invoke-RestMethod `
  -Uri 'http://127.0.0.1:18444/admin/v1/pairing-code' `
  -Headers $Headers
```

模拟门锁卡住和低电量：

```powershell
Invoke-RestMethod `
  -Method Post `
  -Uri 'http://127.0.0.1:18444/admin/v1/devices/door-entry-01/fault' `
  -Headers $Headers `
  -ContentType 'application/json' `
  -Body '{"jammed":true,"batteryPercent":10}'
```

恢复门锁：

```powershell
Invoke-RestMethod `
  -Method Post `
  -Uri 'http://127.0.0.1:18444/admin/v1/devices/door-entry-01/fault' `
  -Headers $Headers `
  -ContentType 'application/json' `
  -Body '{"jammed":false,"batteryPercent":100}'
```

模拟灯离线和恢复：

```powershell
Invoke-RestMethod -Method Post `
  -Uri 'http://127.0.0.1:18444/admin/v1/devices/light-living-01/fault' `
  -Headers $Headers -ContentType 'application/json' -Body '{"online":false}'

Invoke-RestMethod -Method Post `
  -Uri 'http://127.0.0.1:18444/admin/v1/devices/light-living-01/fault' `
  -Headers $Headers -ContentType 'application/json' -Body '{"online":true}'
```

注入环境数据：

```powershell
Invoke-RestMethod -Method Post `
  -Uri 'http://127.0.0.1:18444/admin/v1/environment/inject' `
  -Headers $Headers -ContentType 'application/json' `
  -Body '{"temperatureCelsius":30,"humidityPercent":70,"illuminanceLux":20,"presence":true}'
```

完成故障演示后必须恢复设备，避免后续开发者把注入状态误认为代码问题。

## 十七、性能和稳定性

完整自动化套件会自行启动测试网关，因此先停止手动网关：

```powershell
cd C:\Users\admin\Desktop\openHarmonyProject\VirtualGateway
.\scripts\test_gateway.ps1
```

30分钟稳定性测试：

```powershell
.\scripts\test_stability.ps1 -DurationSeconds 1800
```

目标：

- 1000 条命令成功率不低于 99.5%；
- 命令 P95 不超过 500ms；
- 状态事件 P95 不超过1秒；
- 连续运行30分钟无崩溃、不可恢复断线或明显资源增长。

同机回环性能不能替代真机局域网延迟证据。

## 十八、常见故障速查

| 现象 | 原因 | 解决方式 |
| --- | --- | --- |
| `ProxyError / WinError 10061` | pip 或 curl 使用了已停止的本机代理 | 检查当前 Terminal 的代理变量；健康检查使用 `trust_env=False` 或 `--noproxy "*"` |
| `curl: (60) revocation status is unknown` | Windows Schannel 无法查询演示 CA 的 CRL | 使用 Python/httpx；或 curl 增加 `--ssl-no-revoke`，不要使用 `-k` |
| `Failed to connect` | 网关未运行、防火墙阻止、地址或端口错误 | 保持 `run_gateway.ps1` 运行；检查 `8443` 监听和局域网 IP |
| 证书校验失败 | SAN 不含输入 IP，或 APP 仍打包旧 CA | 按当前 IP 重新生成证书，Clean、Rebuild、重新安装 APP |
| `hdc` 显示 `[Empty]` | 模拟器未启动或真机未连接 | 在 Device Manager 启动 API 24 模拟器，重新执行 `hdc list targets` |
| `12000011: entity does not exist` | 运行了未包含 HUKS 首次建密钥修复的旧 HAP | 更新代码，Clean、Rebuild，卸载旧 APP 后重新 Run |
| 配对码无效 | 超过5分钟、已成功使用、输入错误或连续失败锁定 | 从网关启动日志或本机维护接口读取当前配对码；避免连续尝试错误码 |
| APP 重启后仍要求配对 | 凭据已过期，或网关进程同时重启导致内存会话失效 | 使用配对页记住的地址和当前一次性配对码重新配对；只重启 APP 且凭据未过期时验证 HUKS 恢复 |
| pytest `PermissionError` | 旧缓存由其他 Windows 账户或沙箱创建 | 使用最新 `scripts/test_gateway.ps1`，不要直接依赖旧 `.pytest_cache` |
| `rg` 无法识别 | DevEco Terminal 未安装 ripgrep | 使用最新 `security_scan.ps1`，脚本会自动回退到 PowerShell |
| 端口被占用 | 手动网关与测试网关同时运行 | 测试前停止手动网关；通过 `netstat` 定位监听进程 |
| Native/CMake 编译失败 | SDK、ABI 或 DevEco 路径不匹配 | 确认 API 24 SDK、`arm64-v8a/x86_64` 和 DevEco 安装路径 |

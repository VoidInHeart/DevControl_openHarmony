# DevControl 扩展性视频录制方案

更新日期：2026-07-18

## 一、评估结论

原《扩展性.pdf》对演示边界的判断是正确的：不能用 APP 本地卡片或页面动画冒充设备接入，能力描述、统一控制链路和审计必须是真实的。但 PDF 更接近需求讨论，不是可直接录制的分镜，而且其中“待接入设备列表、一次性 claim code、运行时注册和接入审计”当前尚未实现。

本次视频应展示已经真实完成的“开发时设备扩展”，不宣称“运行时硬件配网”：

> 同一个 HAP 不修改设备类型、解析或页面分支；网关新增并注册一个风扇驱动后，APP 重新同步快照，自动生成风扇分组、卧室设备卡片和开关/枚举控件。后续控制继续复用 HTTPS/WSS、AES-GCM、状态版本和审计链路。

### 1.1 当前真实证据

| 证据 | 当前状态 | 视频呈现方式 |
| --- | --- | --- |
| 独立新设备驱动 | 已完成 `VirtualFanDriver` | IDE 展示驱动文件 |
| 显式注册入口 | 已完成 `composition.py` | 展示唯一注册行 |
| APP 无设备分支修改 | 已完成 | 终端显示 APP 源码改动为 0 |
| 动态功能分组与房间展示 | 已完成 | APP 展示“风扇”和“卧室” |
| 通用开关、枚举控件 | 已完成 | 控制电源、风速和摇头 |
| 权威状态回传 | 已完成 | 操作后等待网关状态更新 |
| 加密控制与审计 | 已完成 | APP 诊断/网关日志展示命令记录 |
| 自动化证据 | 30 项 pytest 通过 | 结尾展示测试结果 |

### 1.2 不应宣称的能力

- 不宣称蓝牙扫描、Wi-Fi 配网、Matter commissioning 或厂商云绑定。
- 不宣称 APP 已有“发现附近设备”页面。
- 不宣称已实现一次性 claim code 或设备出厂证书验证。
- 不把网关重启后的设备出现剪辑成“热插拔”或“无需重启”。
- 不宣称虚拟风扇使用任何真实厂商私有协议。

镜头必须覆盖“代码扩展、APP 自动识别、真实控制回传”三个层次。只展示其中一个层次都不足以证明可扩展性。

## 四、录制前操作清单

### 4.1 固定环境

- 使用同一台模拟器、同一 HAP、同一分辨率完成前后镜头。
- 关闭系统通知、聊天软件浮窗、输入法候选窗和无关终端。
- 设备时间、网关时间和录屏电脑时间保持一致。
- APP 初始停在总控页，字体缩放保持默认。
- 终端使用深色背景、20～24 号等效字号，只保留当前命令和必要日志。

### 4.2 基线镜头

在 `VirtualGateway` 目录启动不含演示扩展的网关：

```powershell
.\scripts\run_gateway.ps1 `
  -WithoutDemoExtensions `
  -InitialPairingCode 123456 `
  -AdminToken extension-video-admin
```

完成配对后拍摄总控和卧室页。此时网关使用五个基础驱动，设备快照中不包含 `fan-bedroom-01`。

### 4.3 扩展镜头

停止基线网关，使用同一 HAP 正常启动：

```powershell
.\scripts\run_gateway.ps1 `
  -InitialPairingCode 123456 `
  -AdminToken extension-video-admin
```

网关重启会使内存会话失效，这是当前设计的真实行为。视频可保留“重新配对”关键动作，但应将等待时间压缩到 2～3 秒，不能剪成热加载。

### 4.4 代码证据

录制前在终端准备以下命令：

```powershell
$appChanges = @(git diff --name-only -- DevControl)
"APP source changes: $($appChanges.Count)"

cd VirtualGateway
python -m pytest -q
```

第一个命令预期输出 `APP source changes: 0`，第二个命令预期输出 `30 passed`。

## 五、镜头与剪辑规范

- APP 画面占主画面 65%～75%，IDE 或终端作为短证据镜头，不要长时间滚动代码。
- 点击前用 0.3～0.5 秒光标停顿，点击后等待权威状态更新再进行下一步。
- 对 `VirtualFanDriver`、`composition.py` 注册行、`30 passed` 使用局部放大或描边框。
- 前后对比使用相同 APP 页面和相同滚动位置，避免评委误以为换了不同版本 APP。
- 每个字幕不超过两行，每行不超过 20 个汉字；关键术语保持一致：能力描述、显式注册、权威状态、统一控制链路。
- 不在视频中展示证书私钥、完整 credential、dataKey、管理令牌或本机签名口令。

## 六、建议旁白全文

> 这里演示的是虚拟网关中的新设备扩展，不是蓝牙或 Wi-Fi 配网。加载扩展前，同一 APP 中没有风扇设备。现在网关只新增一个 `VirtualFanDriver`，声明设备分类、状态和电源、风速、摇头三个控件，并在统一组合入口注册。APP 端没有新增设备类型、解析分支或专用页面。网关重启并重新同步后，APP 自动出现“风扇”功能分组，设备也进入卧室页面。我们依次开机、切换高速并开启摇头；每次页面变化都来自网关的权威状态回传，控制继续使用原有 AES-GCM、WSS、状态版本和审计链路。自动化测试全部通过。这说明常见新设备只需新增并注册驱动，不需要修改 APP 核心。

## 七、失败重拍与应急方案

| 问题 | 现场处理 | 剪辑策略 |
| --- | --- | --- |
| 网关端口被占用 | 使用 `netstat -ano` 确认，不直接结束未知进程；改用预先准备的备用端口和对应证书 | 不保留排障过程 |
| APP 未出现风扇 | 确认没有使用 `-WithoutDemoExtensions`，重新配对并获取全量快照 | 从重新配对前切入 |
| 控件显示但状态不变 | 停止连续点击，检查设备是否在线及命令处理中状态 | 保留一次失败反馈仅用于异常能力片段 |
| 网关重启后回到配对页 | 属于当前真实设计，使用固定演示码重新配对 | 可压缩等待，不可称为热加载 |
| Hvigor 并行构建内存不足 | 使用 `--no-parallel --no-daemon --max-old-space-size 8192` | 构建放在录制前，不在主视频展示 |

## 八、录制验收清单

- [ ] 明确展示 APP 源码改动为 0。
- [ ] 风扇同时出现在功能分组和卧室页面。
- [ ] 电源、风速、摇头三种控件均真实成功。
- [ ] 至少展示一处命令审计或网关处理记录。
- [ ] 展示 30 项测试通过。
- [ ] 全片没有泄露凭据、密钥、令牌或签名信息。

## 九、后续升级路线

如果比赛规则或答辩明确要求“由 APP 点击添加设备”，则应另立功能迭代，实现待接入设备 API、一次性 claim code、注册持久化、APP 发现页和接入审计。完成这些功能前，不应把当前开发时扩展演成运行时 onboarding。
